# sakura-super-agent

这是一个简化的对话系统实现，演示了三个最核心的设计要素：

1. **模型调用**：`streamText + model`
2. **消息管理**：`messages: ModelMessage[]`
3. **交互循环**：`ask()` 递归调用

## 核心概念

### 1. 模型调用（Provider 模式）
模型调用通过统一接口完成：无论后端是 `mock` 还是实际 API，调用方式完全一致。

- `streamText` 负责从模型获取流式响应
- `model` 负责实际调用底层推理服务

这种统一方式就是 Provider 模式的价值所在：

- 模型实现可插拔
- 测试环境和生产环境使用同一套调用逻辑
- 核心业务代码与具体模型实现解耦

### 2. 消息管理（全量上下文传递）
对话上下文由一个消息数组管理：`messages: ModelMessage[]`

每一轮对话：

- push 一条 `user` 消息
- push 一条 `assistant` 消息
- 下一轮把整个 `messages` 数组传给模型

这是一种最基础的上下文管理方式：

- 不做压缩
- 不做截断
- 不做缓存
- 直接全量传递给模型

### 3. 交互循环（递归 ask）
交互逻辑通过 `ask()` 递归实现：

- 读取用户输入
- 调用 `streamText` 获取模型响应
- 把用户/助手消息追加到数组
- 再次调用 `ask()` 进入下一轮

这个循环结构形成了最简单的 `readline → streamText → push → readline` 流程。

## 后续演化方向

这三个要素在后续课程中会逐步演化成更复杂的模块：

- **模型调用** → `StreamConsumer`
  - 解析工具调用
  - 跟踪推理事件
  - 统计 token 用量

- **消息管理** → 四层上下文管理
  - 截断
  - 时间衰减修剪
  - LLM 摘要压缩
  - 缓存优化

- **交互循环** → Agent Loop
  - `while (true) { think → act → observe }`
  - 让模型自己决定调用哪个工具
  - 让模型决定何时停止

## 结构说明

本仓库的实现适合用于学习和扩展。它把最简单的对话引擎作为基础框架，后续可以在此基础上逐步加入更高级的 Agent、消息管理和模型调用能力。
