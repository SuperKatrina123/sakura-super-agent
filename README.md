# sakura-super-agent

这是一个简化的对话系统实现，演示了三个最核心的设计要素：

1. **模型调用**：`streamText + model`
2. **消息管理**：`messages: ModelMessage[]`
3. **交互循环**：`ask()` 递归调用

## 从 ChatBot 到 Agent 的演化

我们把上一篇的 ChatBot 改成了 Agent，代码层面的变化其实不大：

### 代码层面的变化

- **定义了工具**：`description + inputSchema + execute`
- **`streamText` 加了 `tools` 参数**
- **用 `fullStream` 替代 `textStream`**，处理工具调用事件
- **加了一个 `while` 循环**，让模型能够多步执行

### 行为上的质变

但行为上的变化是质的——AI 从"只会说"变成了"能做"。

我们可以来对比一下改造前后的代码结构差异：

- **从单次调用变成了循环调用**
- **从只处理文本变成了处理多种事件类型**
- **实际上就是加了一层"决策"**——模型每一轮都可以选择"继续调工具"还是"直接回复"

### Agent Loop 的心脏

这个 `while` 循环就是 Agent 的心脏。后面所有课程的内容——保险丝、工具系统、上下文管理、流式响应——都是在这个循环里面或者围绕这个循环做文章。

你可以把它想象成一个操作系统的主循环：

- **CPU**：取指令 → 执行 → 写回结果 → 再取下一条指令
- **Agent Loop**：取模型输出 → 如果是工具调用就执行 → 把结果写回 messages → 再调模型

CPU 的退出条件是关机指令，Agent 的退出条件是模型决定"我已经有足够的信息了，可以回复用户了"。

## Agent Loop 的三道防护

本次更新给 Agent Loop 逐层加了三道防护，每加一层都要跑一次，亲眼看效果：

### 🔄 循环检测（短路保护）
**作用**: 模型反复做同样的事且没有进展，检测到并打断它

**实现**: `loop-detection.ts`
- 滑动窗口记录最近 30 次工具调用
- 检测三种循环模式：通用重复、乒乓循环、无进展重复
- 分级响应：警告 → 熔断

**测试**: 输入"测试死循环"，观察模型是否被及时打断

### 🛡️ API 容错（过载保护）
**作用**: API 限流、超时、网络断开，自动重试而不是直接崩

**实现**: 指数退避重试 + 错误分类
- 429 错误：等待后重试
- 5xx 错误：快速重试
- 网络错误：渐进式退避

**测试**: 输入"测试重试"，观察是否自动重试而不是崩溃

### 💰 Token 预算（漏电保护）
**作用**: 累计追踪 token 消耗，超预算自动停止

**实现**: 实时 token 计数器
- 输入 token + 输出 token
- 预设预算上限
- 接近上限时警告，超限时强制停止

**测试**: 长时间对话，观察是否在预算耗尽前停止

### 🏠 配电箱比喻
你可以把它们类比成家里配电箱的三种保护：
- **循环检测** = 短路保护（防止电流乱窜）
- **API 容错** = 过载保护（防止设备过热）
- **Token 预算** = 漏电保护（防止资源泄露）

各管各的，互不干扰，但少了任何一个都不踏实。

## 核心概念

### 1. 模型调用（Provider 模式）
模型调用通过统一接口完成：无论后端是 `mock` 还是实际 API，调用方式完全一致。

- `streamText` 负责从模型获取流式响应
- `model` 负责实际调用底层推理服务

这种统一方式就是 Provider 模式的价值所在：

- 模型实现可插拔
- 测试环境和生产环境使用同一套调用逻辑
- 核心业务代码与具体模型实现解耦

### 2. 消息管理（全量上下文传递）
对话上下文由一个消息数组管理：`messages: ModelMessage[]`

每一轮对话：

- push 一条 `user` 消息
- push 一条 `assistant` 消息
- 下一轮把整个 `messages` 数组传给模型

这是一种最基础的上下文管理方式：

- 不做压缩
- 不做截断
- 不做缓存
- 直接全量传递给模型

### 3. 交互循环（递归 ask）
交互逻辑通过 `ask()` 递归实现：

- 读取用户输入
- 调用 `streamText` 获取模型响应
- 把用户/助手消息追加到数组
- 再次调用 `ask()` 进入下一轮

这个循环结构形成了最简单的 `readline → streamText → push → readline` 流程。

## 后续演化方向

这三个要素在后续课程中会逐步演化成更复杂的模块：

- **模型调用** → `StreamConsumer`
  - 解析工具调用
  - 跟踪推理事件
  - 统计 token 用量

- **消息管理** → 四层上下文管理
  - 截断
  - 时间衰减修剪
  - LLM 摘要压缩
  - 缓存优化

- **交互循环** → Agent Loop
  - `while (true) { think → act → observe }`
  - 让模型自己决定调用哪个工具
  - 让模型决定何时停止

## 结构说明

本仓库的实现适合用于学习和扩展。它把最简单的对话引擎作为基础框架，后续可以在此基础上逐步加入更高级的 Agent、消息管理和模型调用能力。
